# Composer Design (P6-UX-01)

Task-aware one-shot dispatch from abtop's Workspace tab.

This document is the design contract for `P6-UX-01` in
`docs/EXECUTION_BOARD.md`. Read it before scaffolding the composer.

For the deferred-area context (why this is gated behind audit + policy) see
`docs/AGENT_HANDOFF.md` and `docs/LIMITATIONS.md` § *Deliberately Deferred*.

## Goal

Let a user select a `.dw` task in the Workspace tab, prepare a safe brief, and
trigger a **single non-interactive** call to `claude --print` (Claude first)
or `codex exec` / OpenCode equivalents — without leaving abtop and without
injecting keystrokes into any running CLI session.

## Non-Goals

- Direct agent-to-agent chat.
- Keystroke injection into a live Claude / Codex / OpenCode REPL.
- Persistent multi-turn conversation per agent (each dispatch is one-shot).
- Broadcast to multiple agents from one prompt.
- Cloud or team sync of dispatch history.

## Out-of-Scope for v1

- Codex and OpenCode dispatch paths (tracked separately as a follow-up).
- Streaming response display (v1 captures stdout to completion).
- Inline diff or apply-patch view of the response (response is saved to a
  file path; rendering is deferred).

## State Machine

Mirrors the kill-control pattern (request → confirm → act).

```
Closed
  │  d  (with a .dw task selected on Workspace tab)
  ▼
Drafting { task, agent, draft, brief }
  │  Enter
  ▼
PreviewBrief { ... }
  │  Enter            ────── audit dispatch-<agent> requested
  ▼
AwaitConfirm { ..., requested_at }
  │  Enter (within DISPATCH_CONFIRM_WINDOW_SECS = 5)
  │  ── policy check: ControlPolicy.is_dispatch_allowed(agent_cli)
  │     ├─ false → audit blocked → Closed (status: "disabled by policy")
  │     └─ true
  │  ── env check: ABTOP_DISPATCH_DRY_RUN set?
  │     ├─ true  → audit dry-run                    → Done
  │     └─ false → audit confirmed → spawn job
  ▼
Dispatching { task, agent, started_at, job_handle }
  │  background thread emits DispatchResult via mpsc
  ▼
Done { result }            audit sent
Failed { error }           audit failed

Any state, Esc → audit skipped (if past Drafting) → Closed
```

Confirmation window: **5s** (defined as `DISPATCH_CONFIRM_WINDOW_SECS`).
Longer than `KILL_CONFIRM_WINDOW_SECS = 2` because the user is reading a brief
preview, not reacting to a hotkey.

## Brief Format

The auto-context block is the existing per-task handoff entry — reuse the
handoff renderer so the brief preview is identical to what `--handoff`
already produces.

Payload sent to `claude --print` (or equivalent):

```text
# Context (auto-generated by abtop, redacted)
{handoff block for this task}

# Question
{user draft from composer}
```

The auto-context block is **read-only** in the composer. The user only edits
the draft area. This keeps the privacy contract simple: the auto-context has
already passed the existing redaction tests, and the draft text is the user's
responsibility.

## Spawn Pipeline

Single helper:

```rust
fn spawn_dispatch(req: DispatchRequest) -> mpsc::Receiver<DispatchResult>
```

Per-agent command:

| Agent     | Command                  | Stdin                 |
|-----------|--------------------------|-----------------------|
| Claude    | `claude --print`         | brief + draft         |
| Codex     | `codex exec --` (TBD)    | brief + draft         |
| OpenCode  | TBD                      | brief + draft         |

Rules:

- Run on a dedicated background thread (not the tick thread).
- 60s wall-clock timeout. Kill the child on timeout, audit `failed`.
- Capture stdout to a `Vec<u8>`, **cap at 256 KB** to avoid runaway responses.
- Pass output through `collector::redact_secrets` and
  `collector::sanitize_terminal_text` before any persistence.
- Write the full redacted output to
  `{audit_dir}/dispatch/{rfc3339-ts}-{task-slug}-{agent}.md`. The path is
  reported back to the UI; abtop never re-reads it.

`DispatchResult`:

```rust
pub struct DispatchResult {
    pub outcome: DispatchOutcome,       // Sent | DryRun | Failed
    pub agent_cli: String,
    pub task_id: String,
    pub project: Option<String>,
    pub started_at: chrono::DateTime<Utc>,
    pub finished_at: chrono::DateTime<Utc>,
    pub response_bytes: usize,
    pub response_path: Option<PathBuf>,
    pub error: Option<String>,          // sanitized
}
```

## Audit Flow

Reuses `audit::actions::DISPATCH_*` and `audit::outcomes::*` constants added
in `P4-DSP-01`. One event per state transition:

| Transition                          | Action               | Outcome      |
|-------------------------------------|----------------------|--------------|
| Drafting → AwaitConfirm             | `dispatch-<agent>`   | `requested`  |
| AwaitConfirm → Closed (Esc)         | `dispatch-<agent>`   | `skipped`    |
| AwaitConfirm → Closed (policy off)  | `dispatch-<agent>`   | `blocked`    |
| AwaitConfirm → Done (dry-run env)   | `dispatch-<agent>`   | `dry-run`    |
| AwaitConfirm → Dispatching          | `dispatch-<agent>`   | `confirmed`  |
| Dispatching → Done                  | `dispatch-<agent>`   | `sent`       |
| Dispatching → Failed                | `dispatch-<agent>`   | `failed`     |

Audit fields:

- `target_kind = "task"`
- `target_id  = sanitize(task_slug)`
- `project    = sanitize(project_basename)`
- `reason     = sanitize(short status string)` — **never** the draft text or
  the agent response.

## UI Layout

Modal overlay on the Workspace tab, ~70% width × 70% height, centered:

```
┌─ Dispatch task → claude-code ──────────────────────────────────┐
│ Task: Dataset drift guardrails [Ready]                          │
│ Project: ml-pipeline                                            │
├─ Auto context (preview, redacted) ──────────────────────────────│
│ - status: Ready                                                 │
│ - phase: Plan                                                   │
│ - acceptance: 0/3 verified                                      │
│ - dependencies: 0                                               │
│ - suggested agent: implementation agent                         │
├─ Your question / instruction ───────────────────────────────────│
│ > _                                                             │
│                                                                 │
├─ Status ────────────────────────────────────────────────────────│
│ Enter twice to dispatch  ·  Esc cancel  ·  Ctrl+R cycle agent   │
└─────────────────────────────────────────────────────────────────┘
```

Keybindings while composer is open:

| Key       | Action                                                    |
|-----------|-----------------------------------------------------------|
| chars/⌫   | Edit draft                                                |
| `Enter`   | Drafting→Preview→AwaitConfirm→Dispatch (per state)        |
| `Esc`     | Cancel (audit `skipped` if past Drafting)                 |
| `Ctrl+R`  | Cycle target agent (skip agents with policy `false`)      |
| `Tab`     | Future: move focus between draft/preview (no-op v1)       |

Narrow mode (<80 cols): same composer, full-screen instead of modal.

## Config

New `~/.config/abtop/config.toml` keys (already wired in `P4-DSP-01`,
default `false`):

```toml
allow_dispatch_claude   = false
allow_dispatch_codex    = false
allow_dispatch_opencode = false
```

Env:

- `ABTOP_DISPATCH_DRY_RUN=1` — verify the whole flow with no real spawn.

## Open Questions

1. **Response display ergonomics**: do we render the first ~10 lines inline
   under "Status" once Done, or only show the saved file path? *Lean: file
   path only for v1; inline preview is a nice-to-have follow-up.*
2. **Multi-line draft input**: ratatui has no built-in textarea. Options —
   embed `tui-textarea` crate (new dep), roll our own minimal one-line input
   (limits prompt length), or split input across multiple lines with `Enter`
   reserved for confirm. *Lean: minimal multi-line buffer with `Alt+Enter`
   for newline; `Enter` always means "advance state".*
3. **Codex dispatch command**: confirm `codex exec` is the right
   non-interactive entry. May require checking upstream Codex CLI docs or a
   quick `codex --help`. *Resolve before subtask `P6-UX-01 codex + opencode
   dispatch parity` starts.*

## Roll-out

1. Design (this doc).
2. Scaffold `ComposerState` in `app.rs` + open/close keybinding.
3. Composer overlay UI (read-only brief, no spawn).
4. Spawn pipeline for Claude with dry-run only.
5. Wire spawn results into evidence bundles.
6. Codex + OpenCode parity.
7. Mark `P6-UX-01` Done; open a follow-up issue for streaming / inline preview.

Each step is a separate commit and a separate row in
`docs/EXECUTION_BOARD.md` *Handoff Notes* until the task is fully Done.
